# dependency-model.md

> Спільний reference planning-suite. Граф залежностей і послідовність на двох рівнях гранулярності. Споживають: `project-planning` (епік/фіча-залежності, критичний шлях арки), `sprint-planning` (work-type усередині фічі, готовність). Конфіг флоу — у local-context → Planning → Development Flow.

---

## 1. Два рівні залежностей

| Рівень | Гранулярність | Хто використовує |
| --- | --- | --- |
| **Макро** | епік ↔ епік, фіча ↔ фіча | `project-planning` — послідовність і критичний шлях арки проєкту |
| **Мікро** | work-type усередині фічі (Design/BE/Analytics → Client → QA) | `sprint-planning` — готовність «що можна брати зараз» + порушення порядку |

Той самий апарат (DAG, топосорт), різна гранулярність.

---

## 2. Джерела залежностей

1. **Jira issue-links** (основне): за конвенцією FET — `Blocks`-ланцюг (послідовність) + `Relates`-пари (між сутностями однієї фічі). Тягнути через `getJiraIssue` (поле `issuelinks`) per-key.
2. **Назви/нумерація** фіч (`SHOPEX-{епік}.{фіча}`) — групування під епік.
3. **Development Flow** з local-context — типова послідовність work-types у команді (мікро-рівень).
4. **Ввід PM** — ручні залежності, яких немає у Jira (gate-уточнення).

> Якщо залежність є в макеті/концепті, але не оформлена Jira-лінком — підсвітити PM як «розрив графа» (кандидат на оформлення), не вигадувати.

---

## 3. Побудова графа

1. Вузли = епіки/фічі (макро) або work-type-задачі фічі (мікро).
2. Ребра = напрям залежності (A → B = «B після A»).
3. **Топологічний сорт** → лінійна послідовність, що поважає всі ребра.
4. **Критичний шлях** — найдовший ланцюг за сумарним обсягом (SP); його зсув = зсув усього графіка.
5. **Виявлення циклів** — якщо A→B→…→A, підсвітити (граф не може бути виконаний як є).
6. **Висячі/відсутні залежності** — фіча без жодного звʼязку там, де очікується послідовність → флаг.

---

## 4. Work-type DAG (мікро) і правило готовності

Дефолтний флоу (override з Development Flow команди):

```
Requirements → Design → {BE, Analytics} → Client (FE / iOS / Android) → QA → Release
```

**Правило готовності:** downstream work-type **готовий до планування**, коли ВСІ його upstream-передумови досягли порогу готовності.

- Поріг (дефолт, override у Development Flow): статус ∈ {on review, in test, ready for test, done, closed} — тобто етап реалізації пройдено (не обовʼязково «done»).
- **Порушення:** downstream планують у спринт, а upstream нижче порогу → флаг «X раніше за {передумова}» з поясненням; рішення PM (свідоме підтвердження), не жорстке блокування.

Приклад: Design=done, BE=on review, Analytics=in test → Client = **Ready** (можна планувати реалізацію на клієнтах). Якби Analytics=planned → Client у спринті = **порушення**.

---

## 5. Обмеження паралелізму

Скільки фіч/слайсів напрямку команда веде одночасно обмежено платформною capacity (`capacity-model.md`). При розкладі (`critical_path_schedule`) не ставити паралельно більше, ніж тримає платформа — інакше прогноз тривалості буде нереалістично оптимістичним.

---

## 6. Виходи

- **Послідовність** (топосорт) — порядок епіків/фіч або work-types.
- **Критичний шлях** — ланцюг, що визначає тривалість; його елементи позначати окремо (їх зсув найдорожчий).
- **Список готовності** (мікро) — Ready / Blocked (чим і до якого статусу) на кожен кандидат.
- **Список порушень** — спроби планувати downstream раніше за upstream.
- **Розриви графа** — очікувані, але неоформлені залежності.

---

## 7. Якість / застороги

- Не вигадувати залежності — лише з Jira-лінків, Development Flow або явного вводу PM; решта = «розрив, оформи».
- Критичний шлях перераховувати при кожному `replan` — перенос його елемента зсуває всю арку.
- Поріг готовності й сам флоу — з Development Flow команди, не хардкод.
- Рекомендації послідовності маркувати «на підтвердження PM/TL».
